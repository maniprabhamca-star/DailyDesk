Detection, not a file picker

The popup reads the active tab for a PDF — a .pdf URL, an embedded viewer, or a direct link. Competitors open a blank drop zone and make you find the file again.

The zero counter is the pitch

A real byte count from the extension's own webRequest tally, not a slogan. It is the only claim in this category that a sceptic can verify in ten seconds — and the only one an upload-based competitor structurally cannot copy.

Six tools, not fifty-seven

Only tools whose engine already runs client-side and fits the MV3 budget. Office conversions and OCR are deliberately absent — they need a server, and putting them here would break the promise the popup makes.

"Open in DiemDesk" is the traffic mechanism

The honest answer to how does an extension send site visits? — it mostly does not. It hands off when the user wants something the popup cannot do: a multi-file job, a signature, the editor. That is a real but modest funnel, and it should not be oversold.

Same engine, same output

Compression runs the identical code path as the site, so a file compressed in the extension is byte-identical to one compressed on /compress-pdf. No second implementation to keep in sync.

Build reality

What it costs, and where it bites

Estimates are calendar time for one developer already running everything else — not ideal dev-days.

PieceStatusNotesTime
Compression / rotate / split engines REUSE Already framework-free in lib/. Lifts across largely unchanged. ~1 wk
pdf.js + WASM under MV3 RISK MV3's CSP blocks unsafe-eval; WASM needs an offscreen document, and the service worker dies after 30s idle mid-job. This is the part that slips. 2–3 wks
Popup UI + detection NEW Screens above. Small surface, but its own design system at 380px. ~1.5 wks
Context menu + in-page bar NEW Content script, per-origin dismissal memory. ~1 wk
Store review RISK File-handling permissions draw a deeper review. Not schedulable — you wait. 2–4 wks

Roughly 10 weeks of build, then a review queue you cannot schedule around. The extension is genuinely good — the zero-byte counter is the sharpest expression of the whole product, and no upload-based competitor can answer it.

But be clear about what it returns. Both of its revenue paths end at /pricing, and today /pricing collects waitlist emails rather than payments. An extension that drives signups into a page that cannot charge is ten weeks spent widening the top of a funnel with no bottom.

The order that makes this worth building: turn billing on, learn whether six people will pay, then come back and build the thing that sends more of them.
